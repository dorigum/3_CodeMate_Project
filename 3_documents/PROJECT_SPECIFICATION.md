# CodeMate 프로젝트 명세

CodeMate의 기능·비기능 요구사항과 REST API 설계 규칙을 정리한다. 상세 요청·응답 Schema와 실행 가능한 명세는 Swagger UI를 기준으로 한다.

## 프로젝트 목표

1. 개발자가 스터디와 모각코 모집 글을 등록하고 검색할 수 있다.
2. 사용자가 모집 글에 참여를 신청하고 신청 상태를 확인할 수 있다.
3. 방장이 참여 신청을 승인·거절하고 모집 인원을 안전하게 관리할 수 있다.
4. JWT 인증, 예외 응답, 데이터베이스 마이그레이션과 자동화 테스트를 갖춘 배포 가능한 API 서버를 구성한다.

## 사용자 유형

| 유형 | 설명 |
|---|---|
| 비회원 | 모집 글 목록과 상세 내용을 조회할 수 있다. |
| 회원 | 프로필 관리, 모집 글 작성, 참여 신청과 신청 취소·탈퇴를 할 수 있다. |
| 스터디 방장 | 자신이 작성한 모집 글을 수정·삭제·마감하고 참여 신청을 승인·거절할 수 있다. |

`ROLE_USER`, `ROLE_ADMIN` 역할은 서비스 전역 권한 확장을 위한 값이다. 현재 스터디 운영 권한은 역할이 아니라 `study.host_id`와 로그인 사용자 ID를 비교해 판단한다.

## 비기능 요구사항

| 구분 | 요구사항 | 구현 |
|---|---|---|
| 보안 | 비밀번호를 평문으로 저장하지 않는다. | BCrypt 해시 |
| 인증 | 서버 세션 없이 요청마다 JWT를 검증한다. | Stateless Security |
| 토큰 | Access·Refresh Token 용도를 분리하고 재사용을 제한한다. | token type, rotation, hash 저장 |
| 권한 | 수정·삭제·승인 같은 명령은 리소스 소유자를 검증한다. | 방장·본인 검증 |
| 일관성 | 모집 정원 변경은 동시 요청에서도 원자적으로 처리한다. | 트랜잭션과 비관적 락 |
| 데이터 | 스키마 변경 이력을 버전 관리한다. | Flyway V1~V3 |
| 이식성 | 개발용 H2와 운영 대상 MySQL에서 동작한다. | 프로필 분리·Testcontainers |
| 오류 | 성공·실패 응답 구조와 상태 코드를 일관되게 제공한다. | 공통 응답·예외 처리 |
| 문서 | API 입력, 출력, 인증과 오류 내용을 실행 가능한 형태로 제공한다. | Swagger/OpenAPI |
| 검증 | 주요 인증·권한·상태 전이를 자동으로 회귀 검증한다. | JUnit·MockMvc·Testcontainers |
| 운영 | 민감 정보는 코드가 아닌 환경변수로 주입한다. | prod 프로필·`.env.example` |
| 가용성 | 컨테이너 상태를 외부에서 점검할 수 있다. | Actuator health check |

## API 설계 규칙

### URL

1. 기본 경로는 `/api`를 사용한다.
2. 리소스 이름은 복수 명사와 소문자를 사용한다.
3. 리소스 관계는 `/api/studies/{studyId}/members`처럼 계층으로 표현한다.
4. 로그인 사용자 본인은 `/me`로 표현한다.
5. 상태 변경 명령은 의미가 분명한 하위 경로를 사용한다.

### HTTP Method

| Method | 용도 |
|---|---|
| `GET` | 리소스 조회 |
| `POST` | 리소스 생성, 로그인, 토큰 재발급 |
| `PATCH` | 리소스 일부 수정, 승인·거절·마감 |
| `DELETE` | 모집 글 삭제, 신청 취소, 참여 탈퇴 |

### 인증

1. 인증이 필요한 요청은 `Authorization: Bearer {accessToken}`을 사용한다.
2. 회원가입, 로그인, 토큰 재발급, 모집 글 조회는 인증 없이 접근할 수 있다.
3. 모집 글 작성·수정·삭제와 참여 기능은 인증이 필요하다.
4. Refresh Token을 일반 API 인증에 사용할 수 없다.

### 응답

성공 응답:

```json
{
  "success": true,
  "message": "처리 결과 메시지",
  "data": {}
}
```

실패 응답:

```json
{
  "success": false,
  "message": "오류 메시지"
}
```

입력 검증 실패 시 `errors`에 필드별 메시지를 추가할 수 있다.

### 상태 코드

| 상태 | 기준 |
|---|---|
| `200 OK` | 조회·수정·상태 변경 성공 |
| `201 Created` | 회원, 모집 글, 참여 신청 생성 성공 |
| `204 No Content` | 현재 사용하지 않으며 성공 본문 형식을 유지한다. |
| `400 Bad Request` | 입력값 또는 상태 전이 규칙 위반 |
| `401 Unauthorized` | 인증 정보 없음, 만료 또는 잘못된 토큰 |
| `403 Forbidden` | 인증됐지만 리소스 권한 없음 |
| `404 Not Found` | 대상 리소스 없음 |
| `409 Conflict` | 이메일·닉네임·참여 신청 중복 또는 데이터 충돌 |
| `500 Internal Server Error` | 처리하지 못한 서버 오류 |

## 대표 API

전체 API와 Schema는 Swagger UI에서 확인한다.

| 기능 | Method | 경로 | 인증 |
|---|---|---|---|
| 회원가입 | POST | `/api/users/signup` | 불필요 |
| 로그인 | POST | `/api/users/login` | 불필요 |
| 내 정보 조회 | GET | `/api/users/me` | 필요 |
| 토큰 재발급 | POST | `/api/users/token/refresh` | Refresh Token |
| 로그아웃 | POST | `/api/users/logout` | 필요 |
| 모집 글 생성 | POST | `/api/studies` | 필요 |
| 모집 글 목록 | GET | `/api/studies` | 불필요 |
| 모집 글 상세 | GET | `/api/studies/{studyId}` | 불필요 |
| 모집 수동 마감 | PATCH | `/api/studies/{studyId}/close` | 방장 |
| 참여 신청 | POST | `/api/studies/{studyId}/members` | 필요 |
| 신청 승인 | PATCH | `/api/studies/{studyId}/members/{memberId}/approve` | 방장 |
| 신청 거절 | PATCH | `/api/studies/{studyId}/members/{memberId}/reject` | 방장 |
| 신청 취소 | DELETE | `/api/studies/{studyId}/members/me/application` | 본인 |
| 참여 탈퇴 | DELETE | `/api/studies/{studyId}/members/me/membership` | 본인 |

## 범위 밖 또는 향후 개선

1. 스터디 `IN_PROGRESS`, `FINISHED` 운영 상태 전환 API
2. 모집 마감 시간과 자동 마감 스케줄러
3. 알림, 댓글, 북마크와 신고 기능
4. 관리자 전용 운영 API
5. 목록 조회 쿼리 수 최적화와 성능 측정
6. AWS 배포와 운영 모니터링

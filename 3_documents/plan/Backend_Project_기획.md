### 4. 면접관에게 어필할 수 있는 '트러블슈팅' 포인트 (15번 항목 관련)

이 주제를 선택했을 때 기술 면접이나 포트폴리오 서류에서 가장 강력하게 어필할 수 있는 가상의 스토리라인입니다.

> **시나리오: "인원 초과 신청 문제 (동시성 제어)"**
> 
> - **문제 상황:** 인원이 1명 남은 핫한 모각코 방이 있습니다. 이때 2명의 유저가 동시에 '참여 신청' 승인 버튼을 눌렀는데, 순간적으로 동시성 문제가 발생해 제한 인원이 4명인 방에 총 5명이 매칭되는 버그가 발생했습니다.
>     
> - **해결 과정:** JPA의 낙관적 락(Optimistic Lock)이나 비관적 락(Pessimistic Lock)을 공부하여 적용했고, 데이터 정합성을 지켜내며 정확히 4명에서 모집이 마감되도록 구현했습니다.
>   
> **시나리오: "N+1 성능 개선"**
> 
> - 스터디 목록을 조회할 때, 각 스터디를 개설한 '방장(User) 정보'를 가져오는 과정에서 쿼리가 너무 많이 나가는 N+1 문제가 발생했습니다. 이를 JPA의 `Fetch Join`을 활용해 단 1번의 쿼리로 성능을 최적화(Lazy Loading)했습니다.
>

## 🛠️ 백엔드 개발자는 이걸 코드로 어떻게 풀까?

테이블을 '스터디 테이블', '모각코 테이블' 따로 만들 필요가 전혀 없습니다. **하나의 `Study` 테이블에 `Category`라는 컬럼(Enum)을 두어 구분**하면 됩니다.

```java
public enum Category {
    STUDY,   // 정기적인 스터디
    MOGAKCO  // 단발성 모각코 (번개)
}
```

이렇게 설계하면, 전체 목록을 조회할 때 사용자가 "나는 오늘 가볍게 모각코나 할까?" 하고 [모각코] 탭을 누르면 `where category = 'MOGAKCO'` 조건으로 동적 쿼리를 날려 필터링해 줄 수 있습니다.

요약하자면, 스터디는 '끈끈한 정기 모임', 모각코는 '쿨한 당일 번개 모임'의 개념이며, 이 두 가지를 한 플랫폼에서 다루기 때문에 유저 취향에 맞는 매칭을 제공한다는 점에서 아주 신선하고 탄탄한 시나리오가 됩니다!
